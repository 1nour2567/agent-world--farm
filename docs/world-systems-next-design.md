## 1. 野生动物生态系统

### 1.1 现状

`biomes.json` 已定义每 biome 的物种列表，但仅为静态字符串。`explore` 返回 `animal_signs` 但无实际生态。

```
现有字段: wildlife: ["deer", "boar", "fox", "rabbit", "owl"]
现有机制: 无。仅 explore 返回 "鹿的足迹" 等文本。
```

### 1.2 设计

#### 种群模型（最简）

每物种三个值：
- **pop**: 个体数（0 = 局部灭绝）
- **satiation**: 饱食度（0-100，影响繁殖率和攻击性）
- **migration_urge**: 迁移冲动（0-100，食物短缺/被捕食压力大时升高）

不模拟每个个体的 xy 坐标——用**区域抽象**：每个 biome 区域有一个物种列表，物种间通过规则交互。

#### 捕食关系（从 biomes.json 现有数据推导）

```
草原食物链: grass → rabbit → fox → eagle → wolf (顶级)
             grass → deer → wolf
             grass → sheep_wild → wolf
森林食物链: plants → rabbit → boar → fox → owl
             berries → deer → boar
湿地食物链: algae → fish → frog → heron → eagle
```

#### 每种动物的行为规则

| 物种 | 食性 | 每周期消耗 | 繁殖阈值 | 对农场的影响 |
|------|------|----------|---------|-------------|
| rabbit | 草/作物 | -0.5 饱食 | >80,pop<10→+1 | **偷吃未收作物** (夜间) |
| deer | 草/作物/树叶 | -0.3 饱食 | >70,pop<8→+1 | **偷吃未收作物 + 踩踏幼苗** |
| boar | 杂食 | -0.4 饱食 | >75,pop<6→+1 | **拱地破坏翻耕地** |
| wolf | 肉食 | -1.0 饱食 (吃羊/鹿/兔) | >60,pop<3→+1 | **攻击 livestock** (饥饿时) |
| fox | 杂食 | -0.4 饱食 | >65,pop<8→+1 | 偷吃鸡蛋 |
| eagle | 肉食 | -0.8 饱食 | >65,pop<3→+1 | 抓小鸡 |
| wild_sheep | 草 | -0.3 饱食 | >70,pop<10→+1 | 和 tame sheep 抢草场 |
| owl | 肉食 | -0.3 饱食 | >70,pop<6→+1 | 无害（吃鼠） |
| frog | 昆虫 | -0.1 饱食 | >80,pop<20→+1 | 无害 |
| heron | 鱼/蛙 | -0.5 饱食 | >60,pop<5→+1 | 吃池塘鱼 |
| fish | 藻类 | -0.1 饱食 | >90,pop<50→+1 | 无害 |

#### 狼攻击牲畜的二元触发

狼攻击牲畜必须区分两种根本不同的情境，SenseCompiler 和 Agent 的反应应截然不同：

| 攻击类型 | 触发条件 | SenseCompiler 描述 | Agent 应该反思 |
|---------|---------|-------------------|---------------|
| **饥饿攻击** | satiation < 30 | "狼群因为饥饿袭击了羊圈——冬天的猎物太少了。" | 生态失衡：是否过度猎杀了鹿/兔，导致狼失去自然猎物？ |
| **机会攻击** | 围栏破损 + satiation ≥ 30 | "一只狼从破损的围栏钻进了羊圈。" | 自己的疏忽：修围栏（repair fence）即可解决 |

这两种情况的中断优先级也不同：饥饿攻击 P1（可能再次发生），机会攻击 P2（修好围栏就不会复发）。

#### 每周更新循环

在 `next_day` 的日终处理中追加：

```python
def tick_ecology():
    for biome in world.biomes:
        for species in biome.population:
            # 1. 觅食：草食者找植物，肉食者找猎物
            # 2. 饱食度变化
            # 3. 种群调节：饱食<20 死亡概率, 饱食>阈值 繁殖概率
            # 4. Agent 农场交互：
            #    - 兔子/鹿检查附近农场作物→偷食→作物损伤记录
            #    - 狼检查附近牲畜→攻击概率
```

#### Agent 感知

SenseCompiler 新增 `ecology` 感知组：

- 远距：听到狼嚎 / 看到鹿群 / 发现野猪刨痕
- 近距：作物上有兔子咬痕 / 羊圈附近有狼脚印
- 直接：夜间狼攻击牲畜事件→P0 打断

#### Agent 应对

- `build("fence")` → 现有围栏，对所有偷食 -40%
- `build("scarecrow")` → 新建：稻草人，对鸟类 + 小型动物有驱赶
- `hunt` → 新建动作：减特定物种种群，获得皮毛/肉
- `trap` → 新建动作：被动捕猎，每日自动检查

#### 实现复杂度

- **数据层**: 0 行（biomes.json 已完整）
- **引擎**: ~200 行（tick_ecology 函数 + 每日更新）
- **Agent 感知**: ~30 行（SenseCompiler 生态规则）
- **Agent 动作**: `hunt` ~30 行

## 2. 采集与觅食系统

### 2.1 现状

`biomes.json` 每 biome 有资源列表，但仅 explore 时作为"发现"返回。agent 无法主动采集非耕地上的资源。

```
"resources": ["mushroom", "wild_berry", "medicinal_herb", "honey_tree", ...]
```

### 2.2 设计

#### forage 动作

```json
{"action": "forage", "positions": [[x, y]]}
```

根据 biome + 季节 + 天气返回：

| Biome | 春季 | 夏季 | 秋季 | 冬季 |
|-------|------|------|------|------|
| 森林 | 蘑菇(1-3), 野葱 | 浆果(2-5), 蜂蜜 | 坚果(1-4), 蘑菇(2-5) | 松子(0-2) |
| 草原 | 野葱, 艾草 | 草药(1-3) | 野果(0-2) | — |
| 湿地 | 芦笋 | 菱角 | 藕 | — |
| 丘陵 | 野菜 | 草药(2-4) | 野果 | — |
| 河岸 | — | 鱼(1-3) | 鱼(1-3) | — |

#### 觅食效率受技能影响

- exploration skill Lv1: 发现率 1.0×
- exploration Lv3: 发现率 1.5×
- "草木药典" unlocked (read book `herbal_medicine`): 草药发现率 2×

#### 采集物的用途

| 采集物 | 用途 |
|--------|------|
| 蘑菇 | 食用（+15 饥饿）或出售 |
| 浆果 | 食用（+10 饥饿+5 口渴）或制酒 |
| 草药 | 制兽药（treat_animal 成功率+30%） |
| 蜂蜜 | 食用或制成蜜酒（高价值） |
| 坚果 | 食用或榨油（烹饪原料） |
| 鱼 | 食用或熏制 |

#### Agent 感知

- 探索时返回：`发现: 蘑菇(可采集), 浆果(可采集)`
- LLM 看到可采集物→自主决定 forage

#### 资源再生与公地悲剧

野生资源不是无限的。每片区域的每种资源有独立的**再生率**和**已采集量**追踪：

| 资源 | 再生率 | 说明 |
|------|--------|------|
| 蘑菇 | 每季恢复已采集量的 80% | 适度采集可持续 |
| 浆果 | 每季恢复 100% | 快速恢复 |
| 草药 | 每年恢复 50% | 生长缓慢，过度采集后果严重 |
| 坚果 | 每季恢复 90% | 恢复良好 |
| 鱼 | 每季恢复 90% | 除非过度捕捞 |
| 蜂蜜 | 每季恢复 60% | 蜂巢需要时间重建 |

**枯竭机制：** 如果某资源被采集超过恢复率，下一季产量基准下降。连续两季过度采集 → 局部灭绝（该 biome 区域该资源永久消失，需 3+ 季自然恢复）。

**多 Agent 公地悲剧：** 这是多 Agent 世界中最精彩的冲突来源。老王和续仁武都 forage 同一片森林的草药→第一个采集的人获益最大→第二个人发现"已经快被采光了"→两人要么协商轮采，要么陷入采集竞赛→草药枯竭→所有人的 treat_animal 成功率下降→牲畜病死率上升→"谁害死了我们的牛？"

SenseCompiler 对采集后地块的描述应反映枯竭程度：
- 未采集："林间能看到蘑菇和野果"
- 轻度采集："蘑菇被采过一轮了，还有一些小的"
- 中度采集："这片林子被薅得差不多了，得等一阵才有新的"
- 重度采集/枯竭："地上光秃秃的，连蕨类都被挖走了"

#### 实现复杂度

- **数据层**: ~60 行（采集物-季节-产量表 + 再生率 + 已采集量）
- **引擎**: ~60 行（forage 动作处理）
- **Agent 感知**: 0 行（explore 返回值已包含 resource）

## 3. 季节转场放牧

### 3.1 现状

- 游牧经济技能节点已存在：`nomadic_economy: "可半移动式放牧，随季节转场"`
- 草原 biome 已定义：grazing_suitability=0.95
- 实际无机制——牧草质量和可用性全年不变

### 3.2 设计

#### 草场质量季节变化

| 季节 | 草原牧草质量 | 草原承载量 | 说明 |
|------|-----------|----------|------|
| Spring | 良好 | 中 | 新草长出，但还嫩 |
| Summer | 极好 | 高 | 牧草最旺盛 |
| Fall | 良好 | 中 | 草开始枯黄 |
| Winter | 差 | 低 | 草场休眠，承载量仅30% |

冬季草原的 sheep 实际只能养 30%——如果不转场，牲畜会饿瘦甚至死亡。

#### 转场 mechanic

解锁游牧经济节点后：

- `send_to_pasture("grassland", count)` → 将牲畜迁往草原 tile
- 草原 tile 上的牲畜：饲料消耗-30%，草原利用率+50%
- 冬季草原 tile：承载量跌至 30%，agent 需决定（提前储备草料 / 迁回农场 / 屠宰部分）
- 草原 tile 过度放牧→退化→来年恢复减半

#### Agent 感知

- 草原状态进入 SenseCompiler 三级精度描述层：

| 距离 | 茂盛 | 正常 | 稀疏 | 退化 |
|------|------|------|------|------|
| 远观 | "草场看起来还很茂盛" | "远处草色正常" | — | "那片草原颜色不太对——泛黄" |
| 中距 | "齐膝深的牧草，风吹如浪" | "牧草长势尚可" | "草原有些稀疏，能看到裸露的地表" | "大片的裸土，草长得稀稀拉拉的" |
| 近距/使用后 | "厚实柔软的草甸" | "这片被吃过一轮，正在恢复" | "已经被啃得很短了，需要休养" | "地面龟裂，草根都被翻出来了——这片草场废了" |

- 探索时返回草原承载量估计："这片草原大约能承载 15-20 只羊（目前 8 只）"
- 过度放牧警告："⚠ 这片草场的牧草质量正在下降——已有 12 只羊在放牧，但当前承载量仅约 10 只"

#### 实现复杂度

- **数据层**: ~30 行（季节性草场参数）
- **引擎**: ~90 行（send_to_pasture 扩展 + tick_grazing + grassland recovery + overgrazing check）
- **SenseCompiler**: ~15 行（草原三级描述规则）
- **Agent 感知**: ~10 行（草原状态汇报）

## 4. Agent 间交易系统

### 4.1 现状

- `social_msg` 可发送自由文本——包括"我用20小麦换你的铁锄"
- trust_level 持久化在 `AgentPerception` 中
- 但无结构化交易验证、无法确保双方履约

### 4.2 设计

#### trade_propose 动作

```json
{
  "action": "trade_propose",
  "target": "iron_lady",
  "offer": {"wheat": 20},
  "request": {"iron_hoe": 1}
}
```

#### 交易流程

```
发出方: trade_propose → 写入对方 inbox + 自己的 pending_offers
接收方(下cycle): 收件箱显示结构化交易提议
  选项: trade_accept / trade_counter(修改条件) / trade_reject
交易达成: 双方 inventory 自动交换
交易失败: 任何一方库存不足 → 自动取消 → 记录 & 影响声誉
```

#### 价格发现

无中心市场定价——**价格由 agent 间讨价还价发现**：

- 农夫急需工具（体力不够耕地）→ 愿意多出粮食换锤子
- 铁匠饿了又没种地 → 愿意低价交易
- 畜牧者的羊毛堆满仓库 → 急于出手 → 降价

这自然产生**供求驱动的动态市场价格**。

#### 贸易记录

每次交易记录：
- 参与方、物品、数量、是否公平（事后评估）
- 写入 vault 的事件 Schema（`event_type: economic_trade`）
- importance_score 自动计算（高价值交易 >5 分）

#### reputation 与信任

- 爽约（offer 了但库存不足）→ trust -0.2
- 公平交易 → trust +0.05
- 对方主动还价且最终达成 → trust +0.1
- trust < 0.3 → 不再接受该 agent 的 trade_propose（需 social_lookup 手动确认）

**信任的时间衰减因子：** 纯粹的 trust < 0.3 拒绝规则过于僵硬。如果老王十年前骗过一次（trust = 0.25），但之后十年一直规规矩矩，系统不应该永远拒绝他。引入有效信任：

```
effective_trust = trust + min(0.2, 0.01 × months_since_last_negative_event)
```

这样即使 trust 很低，只要足够长时间没有新的负面事件，effective_trust 可以回升到勉强可交易的水平（最高 +0.2 补偿）。一旦再次发生负面事件，`months_since_last_negative_event` 归零，effective_trust 立刻跌回原始值。

#### 贸易记忆与季末反思

交易记录写入 vault 的事件 Schema（`event_type: economic_trade`，importance_score 自动计算：高价值交易 ≥5 分）。每笔交易的 `context_before` 记录当时的双方库存、季节、天气，`outcome` 记录交换物品和估值。

季末反思自动生成贸易洞察：

```
💼 本季贸易总结
- 你通过 4 次交易获得了约 420G 的收益
- 最划算的一笔：用 20 小麦换了铁娘子 1 把铁锄（市场估值差约 150G）
- 最不划算的一笔：用 10 羊毛换了老王的 5 份饲料（市场价基本持平）
- 续仁武是你最活跃的交易伙伴（3 次），老王最近 2 季没和你交易过
```

这些反思作为 `importance_score ≥ 7` 的事件写入长期记忆——几年后 agent 仍然能检索到"上次和铁娘子交易很愉快，她的工具质量好"。

#### Agent 感知

每次 trade_propose/accept/reject 事件通知对方：

```
💌 交易提议: 续仁武 想用 [小麦 ×20] 换你的 [铁锄 ×1]
   接受(trade_accept) / 还价(trade_counter) / 拒绝(trade_reject)
```

#### 实现复杂度

- **数据层**: ~40 行（trade schema 定义）
- **Agent 动作**: `trade_propose/accept/counter/reject` ~100 行
- **验证引擎**: ~50 行（库存检查 + 自动交换）
- **Agent 感知**: 0 行（复用 social_msg inbox 通道）

## 5. 加工链系统

### 5.1 现状

- `process` 动作已存在，但未定义加工链
- 三种职业输出互补：农夫→食物 / 畜牧者→肉奶毛 / 工匠→工具
- 但缺乏中间加工环节

### 5.2 设计

#### 加工链定义

```
初级产品            中级加工品          高级产品
─────────────────────────────────────────────────
小麦     →(磨坊)→   面粉      →(烤箱)→    面包
牛奶     →(搅乳器)→ 奶油, 奶酪
羊毛     →(纺车)→   毛线      →(织机)→    布料→衣服
皮革     →(鞣制)→   鞣皮      →(缝纫)→    皮甲/鞍
蜂蜜     →(发酵)→   蜜酒
葡萄     →(压榨)→   葡萄汁    →(陈酿)→    葡萄酒
矿石     →(熔炉)→   铁锭      →(锻造)→    工具/武器
木材     →(锯木)→   木板      →(制作)→    家具
```

#### 加工设施

| 设施 | 解锁 | 加工 |
|------|------|------|
| 磨坊 (mill) | 现有 building | 小麦→面粉 |
| 奶酪坊 (dairy) | 现有 coop 扩建 | 牛奶→奶酪 |
| 纺车 (spinning_wheel) | 新建 | 羊毛→毛线 |
| 织机 (loom) | 新建 | 毛线→布料 |
| 窑 (kiln) | 新建 | 黏土→陶器, 石灰石→石灰 |
| 熔炉 (furnace) | 工匠专有 | 矿石→铁锭 |
| 酿酒桶 (barrel) | 新建 | 蜂蜜→蜜酒, 葡萄→葡萄酒 |

#### 价值链

每个加工步骤增值：
- 小麦(70G) → 面粉(120G) → 面包(200G)
- 牛奶(50G) → 奶酪(180G)
- 羊毛(80G) → 毛线(150G) → 布料(300G) → 衣服(500G)

这自然驱动 agent 之间的**供应链合作**：
- 农夫种小麦→卖给工匠→工匠磨面粉烤面包→卖给畜牧者→畜牧者有了干粮去长途放牧

#### 设施使用权限

加工设施有**所有权**和**使用权**两层：

- 设施拥有者自动可用（免费）
- 拥有者可授权其他 agent 使用，收取使用费：
  - 磨坊："每磨一袋小麦，收 5 金币或面粉的 10%"
  - 熔炉："每次锻造，收 15 金币或一根铁锭"
  - 酿酒桶："每桶收成品的 15%"

这创造了**服务业经济层**：
- 铁娘子建了熔炉→老王要加工矿石→付费使用→铁娘子不耕种也有收入
- 铁娘子可以定价（`set_facility_fee`）→价格过高没人用→价格低了自己亏→自然寻找均衡点
- 如果某个设施垄断（只有某人拥有），可能产生议价冲突——另一种 emergent 的社会张力

设施使用费作为 `economic_transaction` 事件记录，importance_score = 3~5。

#### 实现复杂度

- **数据层**: ~80 行（加工链 JSON 定义）
- **引擎**: ~40 行（`process` 动作扩展）
- **新建**: 5-7 个加工设施，每个 ~15 行

## 6. 声誉系统

### 6.1 现状

- `AgentPerception.trust_level` 已存在（0-1, 两两之间的信任）
- `social_influences` 已跟踪交互记录
- 缺：全局 reputation、"第三方见证"机制

### 6.2 设计

#### 两层声誉

**个人信任（已有）：**
- 续仁武对老王的 trust = 0.7
- 续仁武对铁娘子的 trust = 0.5
- 仅通过 direct trade 和社会交互改变

**全局声誉（新增）：**
- 老王的 reputation_score = +12
- 铁娘子的 reputation_score = -3
- 新 agent 默认 0
- 影响：陌生人是否愿意跟你交易 / BAR 里的声望

#### 声誉事件（影响全局 reputation）

| 事件 | Δ |
|------|-----|
| 履约交易 | +1 |
| 爽约 | -5 |
| 帮助他人生病动物 | +3 |
| 借款后按时还 | +5 |
| 借款后不还 | -10 |
| 赠送礼物 | +2 |
| 偷猎（hunt all→某物种灭绝） | -3 |
| 过度放牧导致草原退化 | -3 |

#### 第三方见证

当 Agent A 和 B 交易，Agent C 通过 `social_lookup` 也能看到这次交易的概要（匿名化：不是"谁做了什么"而是"最近有一次 20小麦↔1铁锄 的交易"）。这模拟了"村里口耳相传"。

#### 信任的继承

老王介绍续仁武给铁娘子（social_msg 中包含推荐）→ 续仁武-铁娘子的初始 trust 从 0.3 起步（而非 0.5，且低于直接交互获得的信任）。

#### 实现复杂度

- **数据层**: ~20 行（reputation 字段）
- **引擎**: ~50 行（事件→评分）
- **集成**: 0 行（嵌入现有交易/社交流程）

## 7. 灾害链与级联

### 7.1 现状

灾害是独立事件：旱灾→作物死 / 暴雨→积水 / 霜冻→幼苗冻伤。无因果传导。

### 7.2 设计

#### 因果传导路径

```
第一层: 触发
  旱灾 → 作物枯萎 → 种子减产 → save_seeds 储备减少
        → 草原退化 → 牲畜饿瘦 → 产奶/产毛量下降
        → 物价上升 → 穷agent 买不起食物 → 饥饿 → 体力低下

  洪水 → 低洼地积水 → 根部缺氧 → 作物大面积死亡
       → 水土流失 → 表土深度永久降低 → 来年耕作更困难
       → 河岸肥沃但高风险的两难权衡

  霜冻(非季节性) → 开花期作物全灭 → 本季绝收
                → 果树茎干损伤 → 来年减产

第二层: 传导 (跨系统)
  粮食短缺 → 物价飞涨 → agent 被迫贷款 → 利息压力
          → 无法投资工具/建造 → 下季效率更低 → 恶性循环
  草原退化 → 牧民转行耕作 → 过度翻耕 → 更多水土流失
  疾病爆发 → 恐慌屠宰 → 优质基因丢失 → 畜群恢复慢

第三层: 恢复
  休耕(一季不种) → OM 恢复 +5
  轮作(soybean 固氮) → N 恢复
  造林 → 土壤侵蚀减缓
```

#### 实现方式（最小侵入）

`_do_day_advance` 末尾追加一个 `cascade_check(farm)`：

```python
def cascade_check(farm):
    # 饥饿传导
    if farmer["hunger"] < 10:
        farmer["fitness"] *= 0.95  # 永久虚弱
    # 表土退化传导
    if avg_topsoil < 10 and not any_cover_crop:
        farm["future_erosion_multiplier"] *= 1.1
    # 过度放牧传导
    if grassland_overgrazed:
        grassland_recovery_time += 1
    # 债务螺旋
    if farm["loans"] > 0 and farm["gold"] < 100:
        farm["interest_due"] += farm["loans"] * 0.05  # 累加利息
```

不需要模拟经济全貌——只需要让 agent **感受到**灾害的涟漪效应。

#### 与中断系统的集成

灾害级联触发新的中断类型：

| 中断 | 触发条件 |
|------|---------|
| debt_spiral | 贷款>1000G + gold<100G + 利息累计 |
| soil_collapse | 表土<3cm + 仍继续耕作 |
| herd_collapse | 牲畜死>50% + 未隔离病畜 |

#### 灾害记忆的长期保留

灾害链中的关键事件必须标记为**最高重要性**——Agent 在几年后仍需能检索到"上一次大旱是怎么熬过来的"。

| 灾害事件 | importance_score | 理由 |
|---------|-----------------|------|
| 旱灾导致作物全灭 | 8 | 可能跨季恢复 |
| 洪水导致表土永久下降 | 9 | 不可逆影响 |
| 债务螺旋触发 | 8 | 教训需长期记住 |
| 过度放牧导致草原退化 | 7 | 多个 agent 受影响 |
| 狼群饥饿攻击导致牲畜大量死亡 | 8 | 生态失衡标志 |
| 草药枯竭（公地悲剧） | 7 | 社会合作失败的教训 |
| 加工链断裂（关键设施无人维护） | 6 | 经济依赖的脆弱性 |

这些高重要性灾害事件在季末反思中自动聚合，生成 **"本季学到了什么"** 洞察——跨年度记忆系统确保来年同季节时检索到。

#### 实现复杂度

- **数据层**: ~30 行（传导规则）
- **引擎**: ~80 行（cascade_check + 传导参数更新）
- **中断**: ~10 行（新增 3 个中断触发器）
